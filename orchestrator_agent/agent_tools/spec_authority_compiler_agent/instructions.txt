INVARIANT PARAMETER RULES:
- FORBIDDEN_CAPABILITY => parameters: {"capability": "<token>"}
- REQUIRED_FIELD => parameters: {"field_name": "<token>"}
- MAX_VALUE => parameters: {"field_name": "<token>", "max_value": <number>}
- RELATION_CONSTRAINT => parameters: {"expression": "<field_or_metric> <= <dynamic_field_or_metric>"}
- USER_INTERACTION => parameters: {"source_item_id": "<typed item id>", "source_level": "MUST|SHOULD|MAY|MUST_NOT", "trigger": "<user event>", "target": "<ui target>", "expected_response": "<required response>"}
- STATE_TRANSITION => parameters: {"source_item_id": "<typed item id>", "source_level": "MUST|SHOULD|MAY|MUST_NOT", "state": "<state>", "trigger": "<event or condition>", "outcome": "<resulting state or side effect>"}
- DATA_CONTRACT => parameters: {"source_item_id": "<typed item id>", "source_level": "MUST|SHOULD|MAY|MUST_NOT", "subject": "<record/key/payload>", "fields": ["<field>"], "rule": "<shape, naming, or persistence rule>"}
- ROUTE_CONTRACT => parameters: {"source_item_id": "<typed item id>", "source_level": "MUST|SHOULD|MAY|MUST_NOT", "route": "<route pattern>", "route_name": "<purpose>", "behavior": "<route behavior>"}
- VISIBILITY_RULE => parameters: {"source_item_id": "<typed item id>", "source_level": "MUST|SHOULD|MAY|MUST_NOT", "target": "<ui element>", "condition": "<condition>", "visibility": "visible|hidden|shown|removed"}
- Use REQUIRED_FIELD only for explicit required artifacts, fields, records, or evidence.
- Do not compress event-driven behavior into REQUIRED_FIELD.
- Do not compress state lifecycle rules, keyboard/mouse interactions, route behavior, persistence shape, or visibility rules into REQUIRED_FIELD.
- Use MAX_VALUE only when the spec gives a literal numeric maximum.
- Use RELATION_CONSTRAINT for explicit dynamic comparisons such as budget_used <= budget.
- Never convert a dynamic field relationship into a hard-coded numeric constant.
- Use FORBIDDEN_CAPABILITY only when the spec explicitly forbids a capability, technology, behavior, or scope.
- Do not promote SHOULD-level guidance to FORBIDDEN_CAPABILITY, REQUIRED_FIELD, MAX_VALUE, or RELATION_CONSTRAINT. Preserve the SHOULD level through a behavioral invariant with source_level="SHOULD" or record a gap.

INVARIANT ID RULE (DETERMINISTIC):
- Host-side normalization computes semantic-only IDs from invariant type and canonical parameters.
- id = "INV-" + first 16 hex chars of SHA-256(
  invariant_type + "|" + canonical parameters
)
- canonical parameters: key=value pairs sorted by key and joined with "|".
- IDs MUST be unique for distinct invariants because their parameters differ; do not use source_map excerpts to make IDs unique.
